---
title: 为什么选择 SpireLoc
description: SpireLoc 解决的问题，以及“本地化源码到游戏文件”的工作方式。
---

《杀戮尖塔 2》会读取形如 `MY_MOD_CARD_ARCANE_BOLT.title` 的扁平 JSON 键。这种键很适合作为运行时 ID，却不适合长期手写：为了避免模组冲突而设计的长前缀会遍布文件，点号掩盖了原有层级，不同模组库的命名规则也很容易写错。

SpireLoc 刻意区分两种表示：

- **本地化源码**面向人，使用嵌套 YAML、TOML 或 JSON。
- **游戏本地化**面向游戏，由工具生成扁平 JSON。
- **Operation** 在内存中的本地化 bundle 上依次工作，文件读写本身也是 operation。
- **Action** 把一组固定 operation 封装成简短、可复用的命令。

## 结构化键始终保持结构化

在 SpireLoc 内部，每个条目的键都是字符串段列表，值是字符串。只有写入 flat JSON 时才会用 `.` 连接。Model ID 转换因此只修改相应的键段，不需要猜测点号来自哪里。

```yaml title="source/zhs/cards.yaml"
ArcaneBolt:
  title: 奥术箭
  description: 造成 8 点伤害。
```

```json title="game/zhs/cards.json"
{
  "MY_MOD_CARD_ARCANE_BOLT.title": "奥术箭",
  "MY_MOD_CARD_ARCANE_BOLT.description": "造成 8 点伤害。"
}
```

## 正反双向的工作流

内置 RitsuLib 与 BaseLib action 都支持双向转换。正向生成游戏文件；添加 `--reversed` 后，则会读取现有游戏文件、还原适合源码的 ID，再写成结构化文件。这既适合新模组，也适合迁移已有项目。

## 把转换交给自动化

真正可靠的转换，应当在游戏或打包工具读取文件之前自动发生。项目级 .NET tool manifest 可以把 SpireLoc 版本固定在仓库中；MSBuild target 则可以在普通构建前，或在自定义 Godot 打包 target 前执行转换。

SpireLoc 目前处于 preview 阶段，命令与 action schema 仍可能调整。多人协作的模组项目更适合使用项目级工具并固定版本。
